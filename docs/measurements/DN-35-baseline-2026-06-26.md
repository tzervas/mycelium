## 1. MEM-4 static-analysis dup-reduction (today's landed analysis)

The MEM-4 static uniqueness pass (DN-33, landed) already elides provably-redundant RC ops at
*analysis* time — this is the static substrate DN-35 builds the *runtime* reclamation on.

| Metric | Value | Tag |
|---|---|---|
| Corpus terms | 11 | `Exact` (count) |
| Owned `Dup` ops (before elision) | 23 | `Exact` |
| `Dup` ops elided | 21 | `Exact` |
| Dup-reduction ratio | **0.913** (91.3%) | `Exact` count ratio, **corpus-dependent** |
| All semantics-preserved | `true` | machine-checked |
| Increment-2 `rc==1` reuse sites found | 3 | `Exact` |
| Increment-2 reclamation records | 16 | `Exact` |
| All reuse sound | `true` | machine-verified |
| `mycelium-mir-passes` lib tests | 47 passed / 0 failed | — |

Source: `corpus::measure_standard()` / `corpus::measure_mem4_standard()` (the crate's own public
functions), invoked via an **ephemeral** scratchpad probe (path-dep on
`crates/mycelium-mir-passes`); `cargo test -p mycelium-mir-passes --lib`.

**Caveats (load-bearing):**
- The dup counts were produced by an **ephemeral scratchpad binary** — `mycelium-mir-passes` has no
  committed test/bin that *prints* the counts (its in-crate tests only *assert* properties like
  `ratio > 0`). The probe called the crate's unchanged public functions; the numbers are the crate's
  real output, not estimates. **Follow-up:** commit a small reproducible printer (a `--bin` or a
  `cargo test -- --nocapture` reporter) so this baseline is re-runnable on demand (recommended Phase-1
  hardening; tracked).
- `0.913` is the corpus's **own aggregate** over its 11 representative terms. The `Dup` *count* is
  `Exact` (read off the IR); the **aggregate ratio is corpus-dependent**, **not** a general claim about
  arbitrary programs (the crate itself flags this).

## 2. Execution-backend timings (`mycelium-bench`, release)

`cargo run --release -p mycelium-bench --bin bench -- --stdout` (release-gated; refuses a debug build —
`refuse_debug_build`). The honest headline is **interpreter vs AOT-env-machine** (both in-process); the
AOT-env-machine is the path DN-35 reclamation lands in.

Tally over 14 execution-backend cases: **2 wins · 26 speed-losses · 0 correctness-losses · 0 errors ·
14 capability-losses · 14 skips.** The interpreter is the trusted fast baseline; the AOT-env-machine is
**slower than the in-process interpreter on most small kernels today**, winning only on recursive builds.

| Case | interp (ns/call) | aot-env (ns/call) | aot-env verdict |
|---|---|---|---|
| rec-build | 46.0k | **37.2k** | **WIN 1.24×** |
| rec-mutual | 59.7k | **50.7k** | **WIN 1.18×** |
| rec-self | 28.1k | 44.6k | loss |
| rec-fold | 39.4k | 156.7k | loss (4×) |
| bit-literal | 87.9 | 13.6k | loss |
| data-construct | 339.9 | 15.3k | loss |

**Caveats:**
- Ran with `mlir-dialect` **OFF** (default) — the 14 MLIR-dialect rows are skips (`Declared`); the
  MLIR-dialect backend is **unmeasured** here (rerun `--features mlir-dialect`; self-skips if libMLIR
  absent).
- `direct-llvm`/`jit` per-invocation timings are **process-spawn-bound** (M-602/E1) — the harness flags
  these as spawn-dominated, not kernel compute; the `0.00×` ratios are exec overhead on trivial
  kernels, not a meaningful compute comparison. The interp-vs-aot-env (in-process) rows are the honest
  headline.
- Timings are warmup + min-mean via `std::time::Instant` (no criterion); `Empirical`, spread tracked in
  the JSON projection. The LLM-harness section ingested a **synthetic** fixture (labeled `mock`) — not
  real model quality, not treated as a measurement.

## 3. What this baseline tells the DN-35 build

- The **static** analysis already removes 91.3% of owned dups on the corpus — but that is *analysis-time*
  elision, **not** runtime reclamation. DN-35 §9 Increment 1 makes the env-machine actually *free*
  (Q6 = drive), and Increment 2 makes it *reuse* in place. The target is **runtime** alloc/peak-memory
  reduction + **constant-memory** iteration (DN-36), which this baseline does **not** yet measure (no
  runtime heap/RSS counters wired) — adding those counters is part of Increment 1's harness.
- The AOT-env-machine being slower than the interpreter on small kernels today is the **gap DN-35
  targets**: real reclamation + FBIP reuse is what should close it (the Perceus/FBIP "within 10% of C++"
  result is the `Proven`-in-literature aim — `Declared` for Mycelium until measured here).
- **Open follow-up before Increment-1 benchmarking is meaningful:** wire runtime heap-alloc + peak-RSS
  counters into the harness (the dup *count* proxy is `Exact` but is not the same as bytes reclaimed at
  runtime). Until then, runtime-memory claims stay `Declared`.
