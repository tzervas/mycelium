## §1 Problem — forward development is under-specified where the collisions actually happen

The repo already has a strong *concurrent* + *landing* discipline: DN-65 bounds the review/landing
surface (≈1–2k-line scoped PRs); DN-20 bounds the test surface (change-scoping); `CLAUDE.md`
§Concurrent-PR development bounds the collision surface (one isolated worktree per agent, disjoint
ownership, integration-tier close-out); mitigation #6 governs the *reconciliation* direction
(pull-down-before-merge-up, no force-push). What is **not** yet written down as a first-class,
automatable pattern is the *forward* half — how a **single new feature/fix/change** is **designed,
shaped, branched, and merged UP** (working → `dev` → `integration` → `main`) so that:

1. the branching seams are disjoint **by construction** (not discovered mid-flight by conflict);
2. each hop stays **bite-sized** and reviewable;
3. the **auto-generated bulk** (`api-index`, `tero-index`, `lib-index`, rendered docs) — which is
   machine-generated, almost always massive, and appears in *every* API-touching change — does not
   drown the change's own PR diff or serialize every PR behind a regen; and
4. the whole thing is invocable **like a skill/command**, so an agent drives it deterministically
   **without holding the procedure in context** (the maintainer's explicit context/memory-efficiency
   goal).

Three under-specified points, concretely.

**(a) Where do the disjoint seams come from?** DN-65 says "decompose along the existing
collision/ownership boundaries," but for a *new* change those boundaries do not exist yet — the change
*creates* them. Discovering them after the code is written is exactly the mid-flight rework DN-65 wants
to avoid. The missing upstream step is **designing the change's shape first** (its API + component
public/private split), because the shape *is* the seam map.

**(b) The size cap fights the auto-generated bulk.** `docs/api-index/` is regenerated by
`just docs-index`, `docs/tero-index/` by `just tero-index-gen`, `docs/lib-index/` by
`just lib-index-gen` (`justfile` lines 279–291, read `Exact` 2026-07-09) — each a whole-tree
regeneration whose diff can dwarf the source change. `CLAUDE.md` §Auto-generated docs already says the
index is *orchestrator-owned* and *regenerated by the integrating parent* — but it does not say **where
in the branch topology** that regenerated bulk lands relative to the change. If it lands *in* the
change's PR, the ≈1–2k cap is blown and the diff is unreviewable; if it lands *after*, the tree is
transiently stale (a drift-gate red). The missing mechanic is a defined **precursor** slot.

**(c) It is not yet a command.** The pattern lives as prose spread across `CLAUDE.md` + DN-65 + DN-20 +
the skills. An agent must *read all of it* to follow it — the opposite of context-efficient. The
missing artifact is a single parameterized skill that *drives* the pattern so the agent invokes it
instead of re-deriving it.

**(d) The deepest problem — context is the scarce resource, and the flow does not manage it.** The
orchestrator's context budget is "the scarcest resource in a multi-wave swarm" (mitigation #6); intent
is silently lost on compaction (mitigation #8). Yet the status-quo flow asks the model to hold the
*whole* procedure and *all* intermediate state — spec, API, component map, code — in one window at
once. The missing idea is to treat forward development as a **staged pipeline that is also a
context-windowing mechanism**: each stage produces exactly the compact artifact the next stage
consumes, and **prior stages drop out of the working set once consumed**. This is the maintainer's
central ask (2026-07-09) and the organizing principle of this note (§2.1) — everything else
(bite-sizing, precursors, the skill) serves it.

This note fixes (a)–(d) and leaves the *downward* reconciliation to the sibling DN it couples to
(`FLAG-synthesis-1`).

## §2 Recommends — the forward-development pattern

### §2.1 The spec-first staged pipeline (the central organizing principle)

**A change is developed as a five-stage pipeline where each stage is a *bounded working set*: it
consumes the prior stage's compact artifact, produces the next stage's, and then drops out of
context.** This is the core of the note — bite-sizing (§2.2) and the precursor mechanic (§2.3) are
subordinate to it. The stages, in strict order (each *feeds forward* into the next; nothing is
designed before its predecessor exists):

**Stage 1 — SPEC / Design Note first (seeds everything).** Before any code *shape*, write the design
documentation: **requirements · deliverables · Definition of Done · success criteria · user stories**.
This directly extends **house rule #6** (which already mandates user stories + DoD on every
epic/issue/ADR/RFC/DN) — the spec stage *is* that mandate placed at the front of the development flow.
For a change large enough to warrant one this is an actual DN/RFC/ADR; for a small change it is the
issue body's stories + DoD. *Output artifact:* a compact spec (the requirements + DoD + stories) —
**the only thing Stage 2 needs.** *Drops from context after Stage 2 reads it.*

**Stage 2 — PUBLIC API (from the spec).** From the spec's requirements + stories, derive the
**public-facing API**: its inputs, outputs, behaviors, and characteristics/guarantees — the `pub`
items (Rust kernel) or surfaced `std.*` names (`.myc`), each with its honest per-op tag
(`Exact/Proven/Empirical/Declared`) stated up front (house rule #1 — the tag *is* part of the API).
The model needs the spec **in context here and little else**. *Output artifact:* the public API
surface + I/O + guarantees — **the only thing Stage 3 needs.**

**Stage 3 — PRIVATE API (from the public API).** From the now-fixed public surface, derive the
**private-facing API**: the internal operations that implement it, where the `EXPLAIN`-able selection
points sit (house rule #2 — no black boxes). The model needs the public API in context; the *spec has
dropped out*. *Output artifact:* the private API surface.

**Stage 4 — COMPONENT shape (from the API's I/O + behaviors).** With public + private APIs fixed, the
**component decomposition falls out easily** — which components are public vs private, how they
interface. This stage's load-bearing by-product is **the touched-file/crate/dir set** — i.e. the
**ownership seam map**. *Output artifact:* the component/interface map + the touched-file set.

**Stage 5 — CODE (stapled in against the now-fully-defined interfaces).** With the spec, APIs, and
component interfaces all fixed, the implementation is *stapled in* against defined interfaces —
`/dev-workflow` discipline (small auditable steps, a property test for every bound, honest tags). The
model needs the *interfaces* in context, not the reasoning that produced them. *Output:* the code.

**Why this is a context-windowing mechanism, not just an ordering.** Each stage "focuses attention on
the specific components at the specific time they're needed, provides exactly the context to feed
forward into the next step, and then drops the portions no longer needed." Stage 3 does not need the
requirements prose; Stage 5 does not need the user stories — it needs the interfaces. So each stage is
a **small, focused working set**, and the total context *ever live at once* is one stage's inputs +
outputs, **not** the whole procedure + all intermediate state. This is the direct operational answer
to mitigation #6 (protect the scarcest resource — context) and #8 (persist each stage's artifact so it
survives the drop): each stage's compact output is written to the issue/PR/spec (durable), then
released from the window. The result is the maintainer's stated goal — a **lightweight, efficient,
highly-focused, well-reasoned, sound-logic development experience** — and it is *why* the pattern is
usable "like a skill/MCP natively": the skill (§4) hands each stage its inputs and lets the model
release the rest, ideally running each stage as a **separate sub-step/sub-agent** whose only handoff is
the compact artifact.

**Why the pipeline is *also* the forward-branching discipline.** Stage 4's output — the touched-file
set — **is the ownership seam map.** Two changes whose Stage-4 file sets are disjoint branch and merge
in parallel **conflict-free by construction** (the DN-65 disjoint-ownership rule, but *derived up front
from the spec-driven API shape* instead of discovered after the code exists). Where two Stage-4 sets
overlap, the overlap is *visible before a line is written* — resolved by ownership (the shared file
rises to the parent, `CLAUDE.md` §Core file-ownership rule), never by a merge conflict. **Spec-first
staging turns the seam map into a design output of Stage 4, not a merge-time surprise** — and it does
so while keeping each stage's context small.

This is a *methodology + a windowing discipline*, not a new artifact class: Stages 2–4 produce compact
declarations that live in the issue / PR / spec, ratified by review. It composes with `/dev-workflow`
(Stage 5) by placing four focused, context-light design stages *ahead* of the first commit.

### §2.2 Bite-sized changes — and an explicit unit ambiguity to resolve

**Bind working→dev and dev→integration to DN-65's ≈1–2k soft rule** (net diff, source only — see
§2.3 for what is excluded). Decompose a larger change along the seams §2.1 reveals — cohesion wins over
a line count (DN-65 §2.1); never fragment into dozens of trivial PRs; a single logical 2.5k unit stays
one PR. Avoid mega-PRs (`> 5k`) on these two hops.

> **`FLAG-units-1` — unit ambiguity, maintainer to confirm (do not silently assume).** The maintainer's
> brief said "1 to 2,000 character difference." **1–2k *characters* is implausibly small** — roughly
> 15–30 lines — which would fragment nearly every real change and directly contradicts the standing
> DN-65 ≈1–2k-**LOC** soft rule. This note therefore designs against **≈1–2k lines/LOC of net diff**
> (consistent with DN-65), and flags the unit for the maintainer to confirm. If the maintainer truly
> means characters, that is a different (much tighter) regime and DN-65 would need superseding, not
> just extending — so it must be an explicit decision, not an inference (VR-5: I do not upgrade a
> guess to a bound).

### §2.3 Precursor doc/index branches — the key new mechanic

**The size cap excludes auto-generated bulk.** `docs/api-index/`, `docs/tero-index/`,
`docs/lib-index/`, and rendered documentation are machine-generated and near-always massive; counting
them against the ≈1–2k rule would be a category error. Instead they get their **own isolated branch**
that merges into the change's **merge-target immediately ahead of the change** — an *immediate
precursor*. The mechanic, precisely:

**What regenerates, and by which recipe** (read `Exact` from `justfile` 2026-07-09):

| Generated artifact | Regen recipe | Output tree | Drift gate |
|---|---|---|---|
| Agent API index | `just docs-index` (`tools/docgen/code_index.py`) | `docs/api-index/` | `just doc-index` |
| Tero memory index | `just tero-index-gen` (`mycelium-tero` bin) | `docs/tero-index/` | `just tero-index` |
| Lib (`.myc`) index | `just lib-index-gen` (`myc-doc` bin) | `docs/lib-index/` | `just lib-index` |
| Rendered docs / docsite | `just docs` / `just docs-site` (advisory) | `target/docsite/` etc. | n/a (advisory) |

**When it regenerates.** Only when the change actually moved a public API / corpus surface — i.e. when
the change's **shape declaration (§2.1) marks a public-API delta**. A private-only change regenerates
nothing (the drift gates stay green), so no precursor is needed. This is why API-first pays for the
precursor mechanic: the shape already told you whether a precursor is required.

**Who owns it and how it stays conflict-free.** The precursor branch touches **only generated trees**
(`docs/*-index/`); the change branch touches **only source**. They are **disjoint by construction** —
the same guarantee §2.1 gives, applied to the source-vs-generated seam. The precursor is
**parent/integrator-owned** (`CLAUDE.md` §Auto-generated docs: the index is regenerated by the
integrating parent, never hand-merged, never union-merged) — a leaf **never** commits regenerated
index bulk into its own change PR; it **FLAGs** "public API moved, precursor regen needed" up, exactly
as it already FLAGs `CHANGELOG`/`Doc-Index` (`CLAUDE.md` §Concurrent-PR development part 1).

**How it merges just ahead, per hop.** For a change branch `B` targeting tier `T`:

1. `B` is reviewed + green under the change-scoped gates (source only; `/pr-land`).
2. **Immediately before** `B` merges into `T`, the integrator cuts a precursor `P` off the current `T`
   tip, runs the relevant regen recipe(s) for the artifacts `B`'s shape marked, commits **only** the
   `docs/*-index/` delta, and merges `P` into `T`. `P`'s drift gate (`just doc-index` /
   `just tero-index` / `just lib-index`) is now green against `T`.
3. `B` merges into `T` next. Because `P` already refreshed the indices, `T`'s drift gate stays green
   across the pair, and **`B`'s own PR diff contains no generated bulk** — it stays small + reviewable.
4. **The pattern repeats going up.** The same precursor-then-change ordering holds into `integration`
   and into `main`: regenerate → merge precursor → merge change, at each tier.

**Ordering rationale (why precursor *ahead*, not behind).** A drift gate is red whenever committed
`docs/*-index/` lags source. If the change landed first, `T` would be transiently drift-red until the
regen landed — a never-silent red the gate is *designed* to raise (G2). Landing the precursor first
keeps every intermediate tip green: the index is refreshed *in anticipation of* the source it
indexes. The one subtlety — the precursor is regenerated from source that only fully exists once `B`
is merged — is handled by regenerating `P` **against `B`'s reviewed tip** (fetch `B`, run the recipe,
keep only the `docs/*-index/` delta, merge `P` then `B`); the index content is computed from `B` even
though `P` lands one commit ahead of it. This is the single genuinely fiddly part of the mechanic and
§5 stress-tests it honestly.

### §2.4 The `main` mega-PR caveat (the one place bite-sizing does *not* bind)

Bite-sizing binds **working→dev** and **dev→integration**. It does **not** bind
**integration→main**. Because every component + change-set was tested individually through
`dev → integration`, the `integration → main` squash **MAY be large** — that is expected and fine
(`CLAUDE.md` squash-only `main`: a linear, bisectable single commit for the *net* change). The
precursor mechanic still applies at this hop (regenerate the indices ahead of the squash so `main`'s
drift gates are green), but the *source* size cap is lifted for the terminal squash. This is the one
caveat: bite-sizing is a **lower-tier** discipline; the terminal release is curated, not sliced.

### §2.5 Coupling to the down-propagation sibling DN

The forward (up) flow and the down-propagation flow are two halves of one loop:
**work flows up** (working → dev → integration → main, this note) and **squashed `main` flows
back down** (main → integration → dev, the sibling DN, per mitigation #6 — pull-down, plain-merge,
never force-push). The forward pattern **must not re-derive** the downward reconciliation; it consumes
it. The coupling seam this note leaves explicit for the synthesis pass:

- After a squash lands on `main`, the sibling's down-propagation refreshes `integration`/`dev` to a
  clean base; the *next* forward change branches off that refreshed tier (**mitigation #13** — branch
  off the working tier, never `main`), so no forward branch ever builds on a superseded base.
- The **precursor mechanic runs in the up direction only** — the down-propagation carries the
  *already-regenerated* indices down with the squashed `main`, so the lowers do not re-regenerate
  (they receive). The synthesis must state this once so the index is regenerated **exactly once per
  tier-crossing**, not twice.

> **`FLAG-synthesis-1` — sibling DN not yet landed; synthesis pass required.** As of 2026-07-09 the
> down-propagation sibling DN is **not** on `origin/dev` (checked: `git fetch origin dev`;
> `ls docs/notes/DN-*Propagation*` / `DN-*Merge*` → none; `git log origin/dev` tip `e9932234`). This
> note therefore designs the coupling **seam** explicitly (above) but cannot integrate the sibling's
> concrete steps. **A synthesis pass must reconcile DN-96 (up) and the sibling DN (down) into one
> unified up-then-down flow** — resolving (i) where exactly down-propagation hands the refreshed tier
> to the next forward branch, and (ii) the once-per-tier-crossing index-regen invariant above. The
> proposed `/forward` skill (§4) should reference the unified flow once both land.

## §3 Alternatives (considered, not recommended)

**Alt-A — extend `/wave` instead of a new `/forward` skill.** `/wave` already drives partition →
isolate → change-scoped leaf → per-PR review → integration close-out. One could bolt the API-first
step and the precursor mechanic onto it. *Rejected as the primary form* because `/wave` is scoped to
**concurrent, multi-item** waves (an orchestrator fanning out N disjoint items); the forward pattern
must also serve the **single change** (one feature, one fix) with no fan-out. A single change should
not have to invoke the whole wave machinery. **Recommendation:** a **thin `/forward` skill for the
single-change/shape+precursor path** that *composes* `/pr-land` and `/land`, and have `/wave` **call
`/forward`'s precursor step** for the multi-item case — extend-by-composition, not duplication (§4).

**Alt-B — regenerate indices *inside* each change PR (no precursor branch).** Simpler topology (no
extra branch). *Rejected*: it blows the ≈1–2k cap on every API-touching PR, buries the source diff in
generated bulk (unreviewable), and serializes every PR behind a full regen. The precursor branch
exists precisely to keep the change diff small and the regen off the critical review path.

**Alt-C — regenerate indices *only* at `integration` (defer all the way up).** This is close to
today's `CLAUDE.md` §Concurrent-PR-development posture (indices finalized at the integration tier).
*Rejected as the sole rule* because it leaves `dev` transiently drift-red for every API change until
integration — a standing never-silent red that trains agents to ignore the gate. The precursor-per-hop
keeps *every* tier green. (Note: Alt-C remains a legitimate *optimization* for a high-churn `dev` where
per-hop regen is too costly — see §5's honest concession.)

**Alt-D — design code-first, extract the API afterward.** The status-quo default. *Rejected as the
forward discipline*: it discovers the ownership seams *after* the code exists, which is the mid-flight
rework DN-65 and this note aim to eliminate. API-first is strictly the earlier, cheaper place to find
the seams.

## §4 Skill proposal — `/forward` (context-efficient, composed, not duplicated)

**Goal (maintainer's explicit ask): the agent invokes the pattern; it does not hold the pattern in
context.** The skill is the deterministic driver so the procedure lives *in the skill*, not in the
model's working memory.

**Name:** `/forward` — "develop a change forward (up the tiers) the spec-first, staged, precursor-clean
way."

**Parameters:**

- `CHANGE` — the feature/fix/change (an issue id, or a short description).
- `TARGET` — the tier this hop lands on (default `dev`; then `integration`; `main` via `/land`).
- `STAGE` — optional resume point (`spec`/`public-api`/`private-api`/`components`/`code`) so a
  compaction or handoff can re-enter the pipeline at the stage whose artifact is already persisted
  (mitigation #8) without replaying the earlier stages.
- `MODE` — swarm model assignment when the change decomposes into a wave (default Sonnet;
  `CLAUDE.md` §Swarm modes), passed to any spawned agent.

**Design invariant — each stage is a separate, context-light step whose only handoff is a compact
persisted artifact.** The skill runs the five pipeline stages of §2.1 as **distinct sub-steps (ideally
distinct sub-agents)**; the parent holds only the current stage's *input artifact* and receives back
only its *output artifact*, so the intermediate reasoning never accumulates in one window. Each stage
writes its artifact to the issue/PR/spec **before** the next begins (durable across the drop). This is
the mechanism, not a nicety — it is what makes `/forward` "context and memory efficient."

**The stages it automates (deterministic + context-windowed, so the model need neither recall the
procedure nor hold all intermediate state):**

- **Step 1 — SPEC (Stage 1).** Produce/attach the spec: requirements · deliverables · DoD · success
  criteria · user stories (house rule #6). *Persists → issue/DN. Hands forward only the spec.*
- **Step 2 — PUBLIC API (Stage 2).** From the spec, derive the public API + I/O + guarantee tags.
  *Persists → issue/PR body. The spec drops; hands forward only the public API.*
- **Step 3 — PRIVATE API (Stage 3).** From the public API, derive the private API + `EXPLAIN` points.
  *Public API in context, spec gone. Hands forward only the private API.*
- **Step 4 — COMPONENTS + seam map (Stage 4).** From the APIs' I/O, derive the component split +
  **the touched-file set**; from that, decompose into ≈1–2k-LOC units along disjoint ownership seams
  (§2.2, DN-65) and mark any shared file to rise to the parent (`CLAUDE.md` §Core file-ownership rule).
  Enforce change-sizing (source-only; warn, never hard-block, on `> 5k`; `FLAG-units-1` unit).
  *Composes `/wave` when the split is multi-item + concurrent — each unit then re-enters `/forward` at
  `STAGE=code`.* *Hands forward only the seam map + per-unit interface.*
- **Step 5 — CODE (Stage 5).** In one isolated worktree per unit (`/worktree-guard --leaf`,
  mitigation #11), branched off the **working tier** (mitigation #13), staple the implementation in
  against the fixed interfaces — `/dev-workflow` change-scoped gates only (DN-20), batched commit+push
  (durability #5/#9; split commit/push, mitigation #12). *Holds the interfaces, not the design prose.*
- **Step 6 — PRECURSOR regen + merge-ahead (§2.3).** If Stage 2/4 marked a **public-API delta**, cut
  the precursor off `TARGET`, run the marked recipe(s) (`just docs-index` / `just tero-index-gen` /
  `just lib-index-gen`) against the reviewed change tip, commit **only** the `docs/*-index/` delta,
  merge the precursor into `TARGET`, then run the drift gate (`just doc-index` / `just tero-index` /
  `just lib-index`) green. Calls **`/doc-index`** (api-index leg) and **`/tero-refresh`** (if a running
  tero server must pick up the new rows). *Skipped entirely for a private-only change.*
- **Step 7 — Upward PR + review.** Open the change PR into `TARGET` (source-only diff, small), run
  **`/pr-land`** (isolated `/pr-review` agent → patch → reply → merge up). `main` is the terminal
  maintainer checkpoint via **`/land`** (squash-only; the §2.4 mega-PR caveat lifts the size cap here).
- **Step 8 — Couple to down-prop.** After a `main` landing, defer to the sibling down-propagation flow
  to refresh the lowers; the *next* `/forward` branches off the refreshed tier (`FLAG-synthesis-1`).

**Composition (extend, do not duplicate):** `/forward` **owns** only the staged design pipeline
(Steps 1–4) and the precursor mechanic (Step 6) — the *new* material. Steps 5/7/8 **delegate** to the
existing `/dev-workflow` · `/worktree-guard` · `/pr-land` · `/land` · `/doc-index` · `/tero-refresh`.
`/wave` gains **one** addition: a call into `/forward`'s precursor step at its integration close-out,
replacing the ad-hoc "regenerate `docs/api-index/`" line with the defined precursor mechanic.

**What must be built:** a new `.claude/skills/forward/SKILL.md` (thin — parameters + the staged loop +
the change-kind→recipe table from §2.3, with `allowed-tools` scoped to `just`, `git worktree/fetch`,
`gh pr`, and the guard scripts), plus a **one-line addition to `/wave`** pointing its precursor step at
`/forward`. No change to `/pr-land` / `/land` / `/dev-workflow` / `/doc-index` / `/tero-refresh` — they
are consumed as-is. **Context-efficiency win (the point):** the agent invokes
`/forward CHANGE=… TARGET=dev`; the skill walks spec → public API → private API → components → code as
separate focused steps, each holding only its stage's working set, persisting each artifact before the
drop — so the model holds *one stage at a time*, never the whole procedure or all intermediate state.

## §5 Adversarial stress-test + honest trade-offs (VR-5 / house rule #4)

No sycophancy: this pattern has three real costs and at least two failure modes. Stated plainly.

**Trade-off 1 — the precursor ordering is genuinely fiddly (the biggest downside).** The precursor
must be regenerated from the change's reviewed tip, yet land **one commit ahead** of that change
(§2.3). That is a small inversion of the natural order (generate-from-X, then land-before-X) and it is
the part most likely to be gotten wrong by hand — an integrator who regenerates from the *old* `TARGET`
tip lands a stale index, and one who lands the change first flips the drift gate red transiently. It is
**safe only because it is automated** (skill step 5); as a *manual* discipline it is error-prone. This
is the strongest argument that the pattern should ship **as a skill or not at all** — the mechanic's
correctness depends on the driver, and a human following it from memory will occasionally invert it.
*Mitigation:* the drift gate is the never-silent backstop (a wrong ordering shows up as a red gate, not
a silent corruption, G2), so the failure is loud, not dangerous — but it is still churn.

**Trade-off 2 — the staged pipeline's up-front overhead, rigidity, and backtracking.** The five stages
are real work done before a line of code, and they assume a **feed-forward** flow where each stage's
output is stable enough for the next to build on. Two honest failure cases: (i) **exploratory changes**
where the API is *discovered* by prototyping, not designed — you pay to produce a spec/API you then
throw away; (ii) **backtracking** — Stage 5 (code) sometimes reveals that the Stage-2 public API was
wrong, forcing a *return* to an earlier stage whose artifact had already dropped from context. Strict
feed-forward-and-drop is in tension with iterative design: a naive reading would forbid the backtrack
or force a full re-derivation. *Honest resolution:* the pipeline is the discipline for changes with a
**knowable surface** (a new stdlib op, a kernel prim, a documented feature) and is **lower-value for
research spikes** — a `dev`-tier exploratory branch may collapse the stages until the surface
stabilizes (`CLAUDE.md`: `dev` is the working tier, messy OK). And because every stage's artifact is
**persisted** (to issue/PR/spec) before the drop (mitigation #8), a backtrack *re-loads* the earlier
artifact from disk rather than re-deriving it — the drop is from the *live window*, not from the
durable record. So the windowing helps the backtrack instead of blocking it; but the note must not
oversell strict staging as friction-free — iterative changes *will* re-enter earlier stages, and that
is a supported (not a forbidden) path via the `STAGE` resume parameter.

**Trade-off 2b — over-staging a trivial change.** Running five formal stages for a two-line fix is
ceremony that costs more than it saves. *Resolution:* the stages **collapse** for small changes — Stage
1 is the issue's stories+DoD (already required by house rule #6), Stages 2–4 are a sentence each, and
the windowing benefit is nil below some size. The pipeline earns its keep on changes big enough that
holding spec + API + components + code at once would *actually* strain the window; below that it is the
ordinary `/dev-workflow` loop. The skill should detect triviality and skip the ceremony (not yet
specified — an open point for the skill build).

**Trade-off 3 — change-sizing forces awkward splits.** The ≈1–2k rule occasionally cuts across a
cohesive unit — e.g. a public API whose *minimal* honest implementation is 2.5k lines (a parser, a
codec) cannot be split without exposing a half-built API (a never-silent violation: a `pub` surface
that refuses on half its inputs). DN-65 already says cohesion wins over the line count, and this note
inherits that: **when the API-first shape shows the unit is atomic, it stays one PR even if `> 2k`** —
the size rule yields to the totality/never-silent rule (house rules #1/#2 outrank a soft process
target). The pattern must not be read as licensing a fragmented, half-exposed API to hit a line count.
This is a real tension the maintainer should note: *the size cap and the no-half-API rule can conflict,
and the no-half-API rule wins.*

**Failure mode A — precursor/change race in a concurrent wave.** If two concurrent changes both move
the public API and both need a precursor into the same `TARGET`, the two precursors touch the *same*
`docs/*-index/` trees — a collision the disjoint-source guarantee does **not** cover (both touch
generated files). *Resolution:* the precursor is **parent-owned and serialized** — the integrator runs
**one** regen after **both** changes are reviewed, landing a single precursor covering both (the DN-65
"shared file → land sequentially, pull-down between" rule applied to the generated trees). The skill's
step 5 must serialize precursors per `TARGET`, never run them concurrently. This is an added constraint,
not a free lunch.

**Failure mode B — regen cost on a high-churn `dev`.** Per-hop regeneration (`just tero-index-gen` is a
`cargo run` over the whole corpus) may be too slow to run on *every* `dev` merge in a busy wave. Honest
concession: for high-churn `dev`, **Alt-C** (defer index regen to the `integration` tier, accept `dev`
drift-red between-times) is a legitimate cost-driven relaxation — at the price of a standing `dev`
drift-red the agents must be told is expected. The recommended default is per-hop-precursor (every tier
green); the escape hatch is documented, not hidden (G2). The maintainer should pick the default for
`dev` based on measured regen cost — this note does not have that measurement (it would be `Empirical`,
and I have not run it; VR-5).

**Net honest verdict.** The pattern's *value* (disjoint-by-construction seams from API-first, small
reviewable diffs, every-tier-green indices) is real and grounded. Its *cost* (precursor ordering
complexity, up-front design overhead, occasional size/cohesion tension) is also real and concentrated
in the precursor mechanic — which is exactly why the skill (automation) is load-bearing, not optional.
This is **not** a free improvement; it trades human-follow-able simplicity for a machine-driven,
collision-free forward flow.

## §6 User stories

- **As a feature author**, **I want** to develop through staged spec → public API → private API →
  components → code steps where each stage hands the next exactly what it needs and earlier stages drop
  from my working context, **so that** I get a lightweight, focused, sound-logic flow and never hold the
  whole procedure + all intermediate state in one window at once.
- **As a feature author**, **I want** the component stage to fall out of the fixed API and yield my
  touched-file set, **so that** my branch's ownership seams are disjoint from other in-flight work by
  construction and my PRs never collide at merge time.
- **As a reviewer**, **I want** the change's PR diff to contain only source (no regenerated
  `api-index`/`tero-index` bulk), **so that** the ≈1–2k-line diff is actually reviewable and the
  generated churn is off my critical path.
- **As an integrator**, **I want** a defined precursor slot that refreshes the indices immediately
  ahead of each change at each tier, **so that** every tier's drift gate stays green and I never land a
  stale index or a transiently drift-red `dev`.
- **As an agent (context-bound)**, **I want** to invoke `/forward CHANGE=… TARGET=dev` and have it
  drive API-first shaping → decomposition → precursor → upward `/pr-land`, **so that** I follow the
  whole pattern deterministically **without holding the procedure in my context**.
- **As the maintainer**, **I want** bite-sizing to bind the lower hops but *not* `integration → main`,
  **so that** the release squash can be a curated mega-PR of individually-tested pieces without
  artificial fragmentation.
