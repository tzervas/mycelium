# Mycelium Language (`tzervas.mycelium-language`)

TextMate syntax highlighting for [Mycelium](https://github.com/tzervas/mycelium) `.myc` files, in
VS Code and VS Code-compatible editors (Cursor, VSCodium, etc).

## Install (recommended — no Azure account needed)

This extension is not yet published to the Microsoft Marketplace (see below). Install the
committed `.vsix` directly — this works identically in **VS Code** and **Cursor**:

```sh
# from a checkout of tzervas/mycelium
code --install-extension editors/vscode/mycelium-language-0.1.0.vsix
# Cursor uses the same CLI:
cursor --install-extension editors/vscode/mycelium-language-0.1.0.vsix
```

Or, in the editor UI: Extensions view → `···` menu → **Install from VSIX...** → select
`editors/vscode/mycelium-language-0.1.0.vsix`.

To build the `.vsix` yourself instead of using the committed copy:

```sh
cd editors/vscode
npm install
npm run sync-grammar   # refresh syntaxes/ from tools/grammar/ (see "Grammar source" below)
npx @vscode/vsce package
```

## Registry status

The project's chosen registry is **Open VSX** — publishing there is **deferred to the public
release** (tracked in the grammar tooling's distribution notes; see
`tools/grammar/README.md` and, once written, `tools/grammar/DISTRIBUTION.md`). No `ovsx publish`
has been run for this package.

**Microsoft Marketplace publishing is explicitly optional and skippable.** It requires an Azure
DevOps organization + a Personal Access Token, which this repo does not provision. The local-VSIX
install path above works fully without either registry — skip Marketplace entirely if you don't
want an Azure account.

## Features

- TextMate-grammar syntax highlighting for `.myc`: comments, strings (with valid/invalid escape
  distinction), numeric literals (binary `0b…`, ternary `0t…`, byte/hex `0x…`, float), keywords,
  type keywords (`Binary`/`bin`/`Ternary`/`tern`/…), guarantee-strength annotations
  (`Exact`/`Proven`/`Empirical`/`Declared`), operators, and `@tier`/`@forage`/`@std-sys`-style
  attributes.
- Language configuration: line comments (`//` — Mycelium has **no** block comments), bracket and
  auto-closing/surrounding pairs, and identifier word-boundaries.

**Not included in this release:** no language server, no IntelliSense/completions, no
diagnostics, no semantic highlighting — this extension is TextMate highlighting only. (LSP
semantic tokens are tracked separately at `crates/mycelium-lsp`.)

## Grammar source — this copy is generated, never hand-edited

The **canonical** grammar lives at `tools/grammar/mycelium.tmLanguage.json` in the main repo,
generated by `tools/grammar/generate.py` from the L1 lexer's keyword table so it can never
silently drift from the language the compiler actually accepts (G2, RFC-0026 §3.4).
`syntaxes/mycelium.tmLanguage.json` in this extension is a **build-time copy** of that file.

**This copy is drift-gated.** `tools/grammar/generate.py` emits it directly (as a downstream copy)
and `just drift-check` verifies it is byte-identical to the canonical grammar — so a stale
`syntaxes/` copy fails the same gate as a stale root-level grammar (G2). Do not hand-edit either
file — edit the L1 lexer and run `just grammar-gen`, which refreshes both. (`npm run sync-grammar`
remains as a manual convenience for an editors-only refresh, but the drift gate is the authority.)

## Testing

Scope assertions live under `tests/*.myc`, run via
[`vscode-tmgrammar-test`](https://github.com/PanAeon/vscode-tmgrammar-test):

```sh
cd editors/vscode
npm install
npx vscode-tmgrammar-test -g syntaxes/mycelium.tmLanguage.json 'tests/**/*.myc'
```

**Verification status (honest, per house rule #1):** grammar scope assertions are
machine-verified (`Empirical` — pass/fail against the actual TextMate tokenizer). Packaging
(`.vsix` contents) is verified by inspection (`unzip -l`). **Live rendering inside a running VS
Code / Cursor window has not been tested in this environment** — no GUI editor is available here.
That is an `Empirical`-not-`Proven` gap: the grammar tokenizes correctly against the test oracle,
but real-editor rendering (theme color mapping, etc.) is unverified.
